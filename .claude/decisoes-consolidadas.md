# Decisoes Consolidadas — FinApp

- **[2026-02-25]** — Supabase browser client singleton: `supabase` NAO deve ser incluido em dependency arrays de hooks React. Motivo: `createClient()` retorna instancia cached, incluir na dep array causa re-renders desnecessarios. Os 17 lint warnings resultantes sao esperados e nao devem ser "corrigidos".
- **[2026-02-25]** — Removidos 4 arquivos orfaos: `navbar.tsx`, `forecast-chart.tsx`, `card.tsx`, `badge.tsx`. Motivo: nenhum era importado por qualquer arquivo do projeto; navbar foi substituido pela sidebar, os demais nunca foram utilizados.
- **[2026-02-25]** — CSRF protection nas API routes (`/api/ai/analyze`, `/api/import/pdf`) via validacao de Origin header. Motivo: proteger endpoints server-side contra requisicoes cross-origin maliciosas.
- **[2026-02-25]** — PII anonymization no contexto do assistente IA: apenas primeiro nome enviado ao Gemini. Motivo: minimizar dados pessoais enviados a APIs externas.
- **[2026-02-25]** — PDF import valida magic bytes (`%PDF-`) alem do MIME type. Motivo: MIME type pode ser falsificado; validacao por magic bytes e mais confiavel.
- **[2026-02-25]** — CSP (Content-Security-Policy) adicionado via next.config.ts com `unsafe-inline` e `unsafe-eval`. Motivo: Next.js requer esses valores para funcionar; restringir quando possivel em versao futura.
- **[2026-02-25]** — Funcao `isRecurringActiveInMonth` extraida para `utils.ts` e compartilhada entre `forecast.ts` e `daily-flow.ts`. Motivo: eliminar duplicacao de logica.
- **[2026-02-28]** — Reconciliacao de saldo usa `initial_balance_cents` + soma de transacoes para calcular saldo esperado. Motivo: permite detectar divergencias entre saldo registrado e calculado sem exigir historico completo de movimentacoes.
- **[2026-02-28]** — Backfill de `initial_balance_cents` na migration 015: `balance_cents - SUM(transacoes)`. Motivo: garantir que contas existentes reconciliem sem divergencia apos a migration.
- **[2026-02-28]** — Gestao de dividas com simulacao iterativa de payoff (max 600 iteracoes). Motivo: simulacao analitica com juros compostos exigiria formula mais complexa e menos flexivel para cenarios com pagamento extra variavel.
- **[2026-02-28]** — `interest_rate_monthly` como NUMERIC(8,4) no Postgres. Motivo: precisao decimal para taxas de juros; retorna como `number` no JS via Supabase client.
- **[2026-02-28]** — AuditLog como tabela imutavel (sem UPDATE/DELETE via RLS). Motivo: trilha de auditoria deve ser append-only para garantir integridade do historico.
- **[2026-02-28]** — `logAudit()` fire-and-forget (sem await). Motivo: nao bloquear a operacao principal do usuario; erros de auditoria nao devem impactar a UX.
- **[2026-02-28]** — Simuladores 100% client-side, sem persistencia em banco. Motivo: ferramentas educacionais que nao precisam salvar estado; simplifica implementacao e evita complexidade desnecessaria.
- **[2026-02-28]** — Testes E2E com Playwright dependem de usuario de teste manual no Supabase. Motivo: nao ha API publica para criar usuarios no Supabase sem service_role key, e expor essa key no CI e inseguro.
- **[2026-02-28]** — CLAUDE.md reestruturado: estado atual e historico de alteracoes movidos para `.claude/estado-atual.md`, decisoes para `.claude/decisoes-consolidadas.md`. Motivo: conformidade com regras do CLAUDE.md global — CLAUDE.md do projeto deve conter apenas regras, contexto e referencias.
- **[2026-02-28]** — `useMemo` removido do `debt-simulator.tsx` (calculo de savings). Motivo: React Compiler (Next.js 16) nao conseguia preservar a memoizacao manual; calculo direto e equivalente pois o componente re-renderiza apenas com mudanca de input.
- **[2026-02-28]** — `eslint-disable preserve-manual-memoization` no `fetchData` do dashboard. Motivo: useCallback legitimo em funcao de 200+ linhas que o React Compiler nao consegue otimizar; suprimir o erro e a abordagem recomendada.
- **[2026-02-28]** — Simulador FI/RE 100% client-side com 3 cenarios (conservador/base/otimista) e taxa de retirada segura (SWR). Motivo: mesmo padrao dos 3 simuladores existentes — ferramenta educacional sem necessidade de persistencia. Conforme Codex Cap. 10 (Aposentadoria/IF).
- **[2026-02-28]** — Fechamento mensal persistente com snapshot de KPIs na tabela `monthly_closings`. Motivo: permite historico mes a mes, comparacao com mes anterior, e registro de observacoes do usuario. Migration 018 com constraint UNIQUE (user_id, month) e RLS completo.
- **[2026-02-28]** — `budgetDeviation` calculado como `((realDespesas - forecastDespesas) / forecastDespesas) * 100` e `fixedExpensePct` como `(totalRecurringDespesas / totalDespesas) * 100`. Motivo: derivar KPIs a partir de dados ja disponíveis no dashboard sem queries adicionais.
- **[2026-02-28]** — Pagina /historico com componente KpiHistory usando dados de `monthly_closings`. Motivo: visualizar evolucao mensal de KPIs com graficos Recharts (receitas vs despesas vs saldo, saude financeira) e tabela DataTable. Dados ja existiam na tabela — apenas consumo read-only.
- **[2026-03-01]** — Removido `unsafe-eval` da CSP. Motivo: desnecessario em producao Next.js 16; reduz superficie de ataque XSS. `unsafe-inline` mantido por necessidade do Tailwind/Next.js.
- **[2026-03-01]** — Waterfall de queries eliminado no dashboard: 4 queries condicionais (recorrencias, savings rates, provisoes anuais) movidas do segundo batch sequencial para o Promise.all principal com `Promise.resolve({data:null})` como fallback. Motivo: reduz latencia em ~200-400ms no carregamento do mes atual.
- **[2026-03-01]** — Limite de reconciliacao reduzido de 50000 para 10000 transacoes. Motivo: query de 50k registros e perigosa para performance; 10k cobre a maioria dos usuarios. Solucao ideal futura: RPC server-side com agregacao.
- **[2026-03-01]** — Whitelist de tags OFX no parser (`ALLOWED_OFX_TAGS`). Motivo: prevenir ReDoS caso tags injetadas sejam passadas ao regex. Tags validas: TRNTYPE, DTPOSTED, TRNAMT, FITID, NAME, MEMO, etc.
- **[2026-03-01]** — Validacao de senha reforçada no registro: exige >= 8 chars + 1 maiuscula + 1 numero. Motivo: alinhamento com boas praticas de segurança; Supabase Auth valida server-side, mas feedback no frontend melhora UX.
- **[2026-03-01]** — IDs de mensagem do chat substituidos de counter global (`let nextMsgId++`) por `crypto.randomUUID()`. Motivo: counter global pode colidir em SSR e entre re-mounts; UUID e seguro e unico.
- **[2026-03-01]** — Array de insights do dashboard memoizado com `useMemo`. Motivo: logica complexa (15 regras condicionais) era recalculada a cada render; memoizacao evita reprocessamento desnecessario.
- **[2026-03-01]** — Paginacao server-side na pagina de transacoes com PAGE_SIZE=50. Motivo: evitar buscar ate 2000 registros de uma vez; usa Supabase `.range()` com `count: "exact"` para paginacao eficiente. Componente DataTable recebeu suporte generico a paginacao via prop opcional `PaginationProps`.
- **[2026-03-01]** — Componentes `Checkbox` e `Textarea` reutilizaveis em `src/components/ui/`. Input/Select atualizados com `py-2.5` (touch target), prop `optional` e disabled styling. Motivo: padronizar formularios — checkboxes raw inline substituidos por componente consistente com label, helpText e error; textarea espelha interface do Input.
- **[2026-03-01]** — Filtros avancados na pagina de transacoes aplicados server-side (Supabase query) com `buildFilteredQuery()` compartilhado entre fetch paginado e exportacao. Motivo: filtros client-side nao funcionam com paginacao server-side; query centralizada evita duplicacao.
- **[2026-03-01]** — Exportacao CSV com BOM UTF-8 e separador `;` em `csv-export.ts` generico. Motivo: Excel pt-BR interpreta `;` como separador e BOM garante acentos corretos. Utilitario generico (`exportToCsv<T>`) para reuso em outras paginas.
- **[2026-03-01]** — Validacoes inline nos formularios: padrao `errors` Record + `serverError` + `clearFieldError()`. Motivo: feedback per-field melhora UX; coleta todos os erros de uma vez em vez de parar no primeiro; erros de servidor permanecem como banner. Aplicado em 8 formularios.
- **[2026-03-01]** — Dark mode via CSS variables semanticas + next-themes (attribute="class", localStorage). Motivo: evita adicionar `dark:` em centenas de classes; tokens como `bg-card`, `text-on-surface`, `border-border` mudam automaticamente com `.dark`. Graficos Recharts usam hook `useChartColors()`. Toggle em Configuracoes > Geral com 3 opcoes (Claro/Escuro/Sistema).
- **[2026-03-01]** — `try/catch` + toast de erro em `fetchData` e `try/catch` em `fetchInvestments` no dashboard. Motivo: 15+ queries em Promise.all sem tratamento de erro deixavam UI em loading infinito caso qualquer query falhasse. `setLoading(false)` movido para `finally`.
- **[2026-03-01]** — Guard NaN em `toCents()`: retorna 0 para inputs vazios ou nao-numericos. Motivo: `parseFloat("")` retorna NaN que propagaria silenciosamente para o banco de dados.
- **[2026-03-01]** — `try/catch` em `calculateDailyFlow` com retorno de resultado vazio. Motivo: Promise.all com 4 queries Supabase nao tinha tratamento de erro.
- **[2026-03-01]** — CSRF check com fallback de header `Referer` quando `Origin` esta ausente, em ambas as rotas API (`/api/ai/analyze`, `/api/import/pdf`). Motivo: requests com Referer de dominio diferente agora sao bloqueadas mesmo sem Origin; requests sem ambos os headers continuam permitidos (server-to-server) pois a autenticacao Supabase protege.
- **[2026-03-01]** — Skill `/e2e-test` expandido para cobertura completa: Dividas (CRUD + simulador pagamento extra), Metas (CRUD + progresso por saldo de conta), 4 Simuladores educacionais, Assistente IA (envio + resposta streaming), Historico de KPIs, Importacao (verificacao de carregamento). Navegacao atualizada de 8 para 12 links, QA visual de 7 para 14 paginas. Motivo: teste anterior cobria apenas CRUD basico (contas, categorias, transacoes, recorrentes, investimentos); novas funcionalidades ficavam sem cobertura E2E.
- **[2026-03-02]** — Manual do Usuario reescrito para v3.0 cobrindo todas as 12 funcionalidades do sistema. `docs/manual-usuario.md` e a fonte unica de verdade; `docs/gerar-manual.py` foi reescrito para ler o markdown e gerar o `.docx` automaticamente (parser de headings, tabelas, blockquotes, listas). Motivo: manual v2.0 nao cobria Metas, Dividas, Simuladores, Historico, dark mode, filtros avancados, exportacao CSV, saldo inicial e reconciliacao de contas.
- **[2026-03-02]** — Pitch Deck reescrito para v3.0 com 14 slides e narrativa de venda estruturada (problema → barreiras → concorrentes falham → comparativo 12x3 → solucao → 3 modulos → IA → evidencias → mercado → 3 pilares → CTA). `docs/gerar-pitch.py` reescrito do zero com componentes reutilizaveis (stat_card, feature_card, pill, checklist, xlist). Motivo: versao anterior tinha 12 slides sem Metas, Dividas, Simuladores e Historico; narrativa de venda era fraca.
- **[2026-03-02]** — Sidebar refatorada com grupos colapsaveis e scroll. Os 12 itens de navegacao foram agrupados em 4 categorias tematicas (Movimentacoes, Planejamento, Analise, Ferramentas); Dashboard e Configuracoes permanecem standalone. Nav recebeu `overflow-y-auto` + `min-h-0` para habilitar scroll. Grupo do item ativo abre automaticamente por pathname. Modo collapsed exibe lista flat de icones sem grupos. Mobile drawer usa `alwaysOpen` (todos os grupos expandidos). Motivo: sidebar sem scroll impedia acesso a itens em telas baixas; 12 itens flat eram dificeis de escanear.
- **[2026-03-20]** — Infraestrutura de load testing criada em `load-tests/` (Node.js puro, sem deps externas). 30 VUs simultaneos contra Vercel + Supabase. Skill `/load-test` registrada para reuso. Motivo: avaliar escalabilidade sob uso concorrente e medir impacto de otimizacoes.
- **[2026-03-20]** — Reconciliacao de saldo alterada de all-time `.limit(10000)` para janela de 6 meses `.limit(5000)`. Motivo: query mais pesada do dashboard; tradeoff aceito (divergencias > 6 meses nao detectadas) pela melhoria de 76% no p95 da homepage.
- **[2026-03-20]** — 5 componentes do dashboard convertidos para dynamic import (InvestmentSummary, MonthlyClosing, RecurrenceSuggestions, GoalsSummary, DebtSummary). Motivo: reduzir JS bundle inicial e TTI; esses componentes ficam abaixo da dobra.
- **[2026-03-20]** — Cache IPCA (API BCB) movido de memory-only para localStorage com TTL 24h. Timeout reduzido de 10s para 5s. Motivo: API BCB pode demorar e bloqueava secao de investimentos; cache 24h e aceitavel para dado que muda mensalmente.
- **[2026-03-20]** — Auth check duplicada removida do layout.tsx (mantida apenas no middleware). Motivo: layout chamava `supabase.auth.getUser()` apos middleware ja ter feito a mesma verificacao, adicionando ~100-200ms por page load.
- **[2026-03-22]** — Transferencia entre contas: 1 linha por transferencia (type='transferencia', destination_account_id, category_id=null). Motivo: evitar duplicacao de receita/despesa em pagamentos de fatura e movimentacoes entre contas proprias. Transferencias sao ignoradas em calculos de receita/despesa, forecast e fluxo diario.
- **[2026-03-22]** — Queries Supabase com hint explicito `accounts!account_id` e `accounts!destination_account_id` nas tabelas transactions e recurring_transactions. Motivo: ambiguidade de FK apos adicionar destination_account_id referenciando mesma tabela accounts.
- **[2026-03-23]** — PDFs com senha: dois caminhos na API de importacao. Path A (com senha): `pdfjs-dist` abre com password, extrai texto, envia texto ao Gemini via prompt dedicado. Path B (sem senha): `pdf-lib` ignoreEncryption (owner-password) + bytes PDF ao Gemini. Motivo: Gemini nao aceita senha de PDF; pdfjs-dist (Mozilla, ~3MB) e a unica lib JS que suporta user-password em ambiente serverless. pdf-lib so resolve owner-password (restricoes de edicao).
- **[2026-03-23]** — Prompts de extracao refatorados com constante compartilhada `EXTRACTION_FORMAT_AND_RULES`. Motivo: EXTRACTION_PROMPT (PDF) e TEXT_EXTRACTION_PROMPT (texto) tinham regras duplicadas; constante unica evita divergencia futura.
- **[2026-03-23]** — Auth defense-in-depth restaurado no layout.tsx do dashboard (getUser + redirect). Motivo: remocao anterior reduzia seguranca — se middleware falhar, rotas ficam desprotegidas. Custo aceito (~100ms extra por page load) pela camada adicional de protecao.
- **[2026-03-28]** — Agrupamento de contas via campo `account_group` (TEXT nullable) na tabela accounts, sem tabela separada. Motivo: solucao leve para separar contas PJ/PF/etc sem CRUD extra; free-text com autocomplete via datalist; contas sem grupo aparecem como "Geral"; UI agrupada so aparece quando ha 2+ grupos distintos.
- **[2026-03-28]** — Relatorio consolidado por grupo como modal na pagina Contas (nao como pagina nova). Motivo: evitar mais um item no sidebar; o relatorio e contextual ao grupo; modal 4xl comporta cards + grafico + tabela; transferencias excluidas dos calculos.
- **[2026-03-28]** — Dashboard refatorado: page.tsx de 754 linhas para ~170 linhas de composicao. Logica extraida em `useDashboardData` (hook principal) e `useInvestmentData` (hook independente). Componentes extraidos: `DashboardHeader`, `RecentTransactions`. `SectionErrorBoundary` adicionado em cada secao para isolar falhas. Layout shell com padding reduzido no mobile. Motivo: dashboard era ilegível em telas < 640px — botoes transbordavam, tabela sem scroll, grafico fixo, grid pulava de 1 para 5 colunas sem transição.
- **[2026-04-19]** — Lucro PJ tratado como **receita líquida consolidada** na visão PF, com drill-down para bruto/impostos/custos. Grupo de categoria pode ser marcado como `is_net_revenue_block`; receitas − despesas do grupo entram como receita única no KPI de receitas PF, e despesas do grupo ficam fora do KPI de despesas PF (evita dupla contagem). Categoria `PF | Lucro PJ` do seed será removida (redundante — o lucro é calculado, não lançado). Plano em `.claude/plans/receita-liquida-pj.md`. Motivo: alinhar o FinApp ao modelo mental de finanças pessoais — a PJ é fornecedora de renda para a PF; a visão consolidada reflete a situação financeira real, o drill-down permite avaliar carga tributária e custos operacionais.
- **[2026-04-19]** — Competência contábil por transação desacoplada da data: novo campo `transactions.competency_month` (TEXT YYYY-MM, nullable). Quando null, competência é derivada de `date` + `closing_day`; quando preenchido, sobrepõe. Reconciliação e fluxo diário continuam usando `date` (são por saldo e por movimento real). Motivo: permitir registrar uma conta de abril paga em atraso em maio mantendo-a na competência correta — importante para análises mensais fiéis ao regime de competência.
